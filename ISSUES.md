- [x] Auto mode silently degrades to Approval-gating terminal commands on hosts without filesystem confinement (landlock reports `ENOSYS` in most containers/VMs, so `Capabilities().FSWrite == false`). The gating itself is **correct** ‚Äî ADR 0012's "confine if you can, gate if you can't" (`resolveLadderAuto`, `internal/agent/resolution.go`) ‚Äî so this is **not** a resolution defect. The defect is that nothing tells the user: `internal/agent/loop.go` refuses Auto only on a *nil* `Confiner`, so a present-but-incapable backend enters Auto with no startup notice, and every terminal command then prompts with no explanation. Needs a startup notice naming the active backend, its capabilities, and the `confine-to-workspace: false` opt-in (the sanctioned "I am the sandbox" path for disposable VMs) ‚Äî **and an offer to apply that setting in place**, so the user doesn't have to go find `~/.apogee/config.yaml` to get the Auto they asked for. The offer is a real reduction in enforcement, so it carries constraints (truthful blast-radius wording, no default-yes, session-scope offered before persistence, visible/reversible config write, and an open question about global-vs-host scope). ‚Äî **closed 2026-07-21**: shipped as a capability-aware startup notice (Auto + confinement asked for + `FSWrite == false`) naming the backend and the consequence, the `/confine` command (`status` / `off` / `off --save` / `on`) as the sanctioned in-place accept path, and a host-scoped `unconfined-hosts:` acknowledgement ‚Äî the open scope question resolved *host-scoped*, so a throwaway container's acknowledgement never follows `~/.apogee/config.yaml` onto a laptop ‚Äî written by a comment-preserving config writer. The **ladder is untouched**: the tool never loosens on its own initiative, it only makes the user's own decision reachable and hard to misunderstand. ‚Üí ADR 0012 (amendment 2026-07-21) and `docs/plans/auto-confinement-degradation-plan.md`.
