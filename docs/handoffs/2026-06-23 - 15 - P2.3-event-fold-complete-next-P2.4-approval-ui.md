## What P2.3 built that P2.4 builds on (all in `internal/tui`, ADR-0010 grep stays empty)

- **`transcript.go`** — the fold is done. New helpers P2.4 can **reuse**: `formatToolCall` and
  **`prettyJSON(json.RawMessage)`** (indent via `json.Indent`; empty/`null` → `""`; malformed →
  verbatim). The pending `ApprovalRequest.Arguments` is a `json.RawMessage`, so P2.4 should render
  it with `prettyJSON` for consistency with the tool-call entries — don't write a second formatter.
- **`ApprovalEvent` is already recorded observationally** (a `· approval <decision>: <tool>` note).
  The loop emits it around the synchronous `Approve`, so once P2.4's key sends the decision and the
  Step continues, the transcript trail of the decision appears **for free** — P2.4 renders the
  *prompt*, not the record.
- **The `debug` field on `transcript`** gates `MechanismFiredEvent` (default off, no product toggle
  yet). If P2.4/P2.5 wants a debug keybinding, flip `m.transcript.debug` — the gate is wired.
- **`renderEntry`** indents continuation lines of a multi-line body at `Depth > 0`; depth-0
  rendering is byte-identical to P2.2 (so the existing golden/substring tests still hold).

## Next: P2.4 — the Approval UI (plan §4 P2.4; ADR 0004; the C3 face)

**Most of the wiring already exists in `model.go` — P2.4 is small.** What is already done:

- `Update`'s `approvalReqMsg` case **already** sets `state = stateAwaitingApproval` and stores
  `m.pending = &msg` (model.go ~L122). The Reply chan is buffered cap 1 (messages.go), so replying
  never blocks.
- **Cancel-clears-prompt already works structurally:** `esc` while `busy()` (busy includes
  `awaitingApproval`) calls `stopWorker()` → the `uiApprover` returns `ctx.Err()` → Step returns
  `StatusCancelled` → `cancelledMsg` → `finishWorker` clears `m.pending`. So the cancel path needs
  a *test*, not new code.
- `hintLine()` already shows `a allow · d deny · s allow-session · esc cancel` in the awaiting state.

**What P2.4 must add (two small things):**

1. **Render the pending request** in `View()` when `state == stateAwaitingApproval`: show
   `m.pending.Request` — the **tool**, the **`Reason`**, and the **`Arguments`** (via `prettyJSON`).
   P2.2's `View` stacks `viewport · status · input · hint`; insert the prompt block (above the input
   or as an overlay on the viewport — a simple stacked block is enough for v1). Only the top-level
   (`Depth == 0`) prompt is handled this phase.
2. **Handle `a`/`d`/`s` in `handleKey` while `awaitingApproval`** → send the decision on
   `m.pending.Reply` (non-blocking) and return to `running`, clearing `m.pending`. **Watch the
   current fall-through:** today, in any non-idle state, unmatched keys go to
   `m.viewport.Update(msg)` (scroll) — so `a`/`d`/`s` are currently swallowed as scroll keys. Add an
   `awaitingApproval` branch *before* that fall-through. Map: `a` → `domain.ApprovalAllow`, `d` →
   `domain.ApprovalDeny`, `s` → `domain.ApprovalAllowForSession`.

**Acceptance (plan §4 P2.4, all under `-race`):** an `approvalReqMsg` puts the model in
`awaitingApproval` and renders the request (tool/args/Reason); each key yields the right
`ApprovalDecision` on `m.pending.Reply` (table test); a stop key while pending clears the prompt and
returns to idle. `allow-for-session` being *observably distinct* (the loop's `approved[...]` cache
suppresses the second prompt) is proven later in the **P2.6** e2e, not here.

Then **P2.5** (config + sessions / snapshot-on-quit) and **P2.6** (hermetic httptest e2e + the
live-model run). The seam + skeleton + fold are there for both.
