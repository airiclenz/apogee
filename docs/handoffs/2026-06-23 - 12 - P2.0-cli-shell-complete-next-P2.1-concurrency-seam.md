## Decisions made in P2.0 (not obvious from the diff)

1. **`internal/tui` holds only the seam boundary so far** — `Engine` (the narrow C5 interface,
   satisfied by `*apogee.Agent`), `Options`, and a **placeholder `Run`** that prints a notice and
   returns nil. The `teaSink` / `uiApprover` / worker were deliberately **kept in P2.1** as the plan
   slices them.
2. **The construction chicken-and-egg is deferred to P2.1, with a labeled placeholder.** `apogee.New`
   requires `Config.Events != nil`, but the real event bridge needs the `*tea.Program`, which does
   not exist until the TUI starts. P2.0 therefore wires a temporary **`nopSink`** (in
   `cmd/apogee/wire.go`) just to satisfy construction. **P2.1 must replace this** (see next section).
3. **The `launcher` seam is injectable** (`cmd/apogee/root.go`): `type launcher func(ctx,
   tui.Engine, tui.Options) error`, defaulting to `tui.Run`. Tests pass a fake to prove clean
   construction + clean quit without a terminal. **This signature will likely change in P2.1** to
   carry the late-bound sink/approver (or bind them another way) — it is an internal seam, change it
   freely.
4. **State roots (C7) are paths-only — no directory creation.** `resolveRoots` computes
   `~/.apogee` (or `--config`) → config/library/sessions and cwd (or `--workspace`) → workspace, all
   absolute. Creating dirs is deferred to the session writer in **P2.5**.
5. **`--resume` was wired early** (it rides the already-stable Session API) with round-trip +
   future-version (`ErrSessionVersion`) tests. **Snapshot-on-quit and the optional `config.yaml`
   remain P2.5.**
6. **Mode handling (C8):** `--mode auto` parses fine but `apogee.New` returns `ErrAutoUnavailable`,
   which `friendlyConstructErr` maps to a Phase-3 message (non-zero exit, no panic).

## Next: P2.1 — the concurrency seam (the hard part)

Build C1–C5 from plan §3 as a cohesive, **rendering-free, `-race`-proven** unit *before* any pane
exists. Concretely:

- **Replace `nopSink`** with the real **`teaSink`** (C2): `Emit(e)` → `p.Send(eventMsg{e})`. Lossless
  default; document a coalescing (never dropping) hook for later. cmd must build the real sink, put
  it in `Config.Events`, call `apogee.New`, then hand the program-binding to `tui.Run`.
- **Add `uiApprover`** (C3): the request/reply rendezvous over a **buffered** `reply` channel, with
  `select { case d := <-reply / case <-ctx.Done(): return ApprovalDeny, ctx.Err() }`. This is the
  single most race-prone piece — it carries the heaviest test (concurrent Emit + Approve + cancel
  under `-race`; verify no goroutine leak when ctx cancels before the UI replies).
- **Worker `tea.Cmd`** (C1/C4): one worker at a time `Submit`s then loops `Step` to the Exchange
  boundary (mirror `coreagent.Run`), holds the `CancelFunc`, returns a terminal `Msg`
  (`exchangeDoneMsg` / `cancelledMsg` / `errMsg`). The worker is the **only** caller of `Engine.*`.
- **The binding mechanism for the chicken-and-egg:** the sink/approver should be **late-bound** to
  the `*tea.Program` created inside `Run` (a `programSender interface { Send(msg any) }` — note
  `*tea.Program.Send(tea.Msg)` satisfies it because `tea.Msg = any`, which also gives the **stub
  programSender** the P2.1 acceptance wants for terminal-free tests).
- **Add deps:** `bubbletea/v2 v2.0.7` (fallback v1 `v1.3.10`); decide v2-vs-v1 at the first widget
  need in **P2.2**, not now. `go get` it when P2.1 starts.
- **Write ADR 0011** ("the TUI is a thin event-driven renderer over a worker-goroutine engine") when
  P2.1 lands — same precedent as Phase 1 → ADR 0007.

Then panes fan out: **P2.2** (Model/Update/View skeleton, Lipgloss/Bubbles), **P2.3** (event
rendering per C6 — note the **8-variant** switch incl. `StreamResetEvent` discard), **P2.4** (the
approval UI), **P2.5** (config + sessions), **P2.6** (hermetic httptest e2e + the live-model run).
