## Three judgment calls a reviewer should ratify or reject

1. **Streaming lives on the concrete `*Client`, NOT on the `Responder` seam.** The loop still
   calls non-streaming `Respond`; the seam stays one method so a fake answers one method.
   Wiring `Stream` into the loop is explicitly **P1.2**'s job (it settles the §6 #6
   streaming+Approval interleave and §6 #3 event-backpressure calls). Reversible — P1.2 can
   widen the seam. **This is the main forward-looking call.**
2. **Exhausted-retry error surfaces the final HTTP status** (`upstream HTTP 502: <body>`)
   rather than a synthetic "after N attempts". More informative (you see the real server
   reply); the retry count is asserted via the call counter in tests. The "after N attempts"
   wrapper only fires for transport/timeout faults (no response to surface).
3. **`ServerManager` is the embeddable-core slice only** — detect / adopt-or-spawn-and-wait /
   stop. PID-file persistence + cross-restart orphan adoption (a VS Code lifecycle concern)
   and the ollama/llama.cpp `/props` discovery strategies are deliberately deferred (the bench
   owns server lifecycle; `/props` runtime-context belongs with the context phase). Noted in
   code. Ratify the scope or pull more forward.
