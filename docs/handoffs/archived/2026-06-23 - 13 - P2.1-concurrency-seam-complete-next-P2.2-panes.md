## What P2.1 built (the seam P2.2 consumes)

All in `internal/tui`, driven only by the narrow `Engine` interface (no root import; ADR-0010 holds):

- **`messages.go`** — the five worker→`Update` messages: `eventMsg{Event}`,
  `approvalReqMsg{Request, Reply}`, `exchangeDoneMsg{Result}`, `cancelledMsg{Result}`,
  `errMsg{Err}`. Each is a plain struct (a valid `tea.Msg`). **P2.2's `Update` folds exactly
  these five + keypresses — nothing else.**
- **`bridge.go`** — `Bridge` holds the `teaSink` + `uiApprover` over one late-bound, **atomic**
  `programRef`. `NewBridge()` → `Sink()` / `Approver()` (installed in `Config` by `cmd`) and
  `Bind(programSender)` (called by `Run` once the program exists). `programSender interface {
  Send(tea.Msg) }` is the single seam a stub stands in for.
- **`sink.go`** — `teaSink.Emit(e)` → `programRef.send(eventMsg{e})`. Lossless; the coalescing
  (never-dropping) hook is documented, **not built** (don't pre-optimise).
- **`approver.go`** — `uiApprover.Approve(ctx, req)` sends `approvalReqMsg` with a **buffered
  (cap 1)** reply chan, then `select { reply / ctx.Done }`. Cancel ⇒ `ApprovalDeny`+`ctx.Err()`;
  the buffer absorbs a late UI reply ⇒ no goroutine leak.
- **`worker.go`** — `startExchange(parent, eng, input) (tea.Cmd, CancelFunc)`; `driveExchange`
  `Submit`s then `Step`-loops to the boundary (`StatusTurnComplete` ⇒ continue), returning one
  terminal Msg. It is the **only** caller of `Engine.*` (single-goroutine contract by construction).

`cmd/apogee` retired the P2.0 `nopSink`: `wire.go` builds the `Bridge`, installs `Sink()`+`Approver()`
in `Config`, and the `launcher` seam now carries `*tui.Bridge`. `tui.Run`'s signature is now
`Run(ctx, eng, br *Bridge, opts)` — **its body is still a placeholder** (prints a notice, binds
nothing) until P2.2 creates the `Model` + `*tea.Program`.

## Two handoff-12 premises were wrong — both corrected (and in ADR 0011)

1. **Bubble Tea module path moved.** The plan pinned `github.com/charmbracelet/bubbletea/v2
   v2.0.7`, but Bubble Tea renamed its module to **`charm.land/bubbletea/v2`** *exactly at v2.0.7*
   (the `github.com/...` path resolves only through v2.0.6). Took the new canonical path
   (best-for-the-future). **Fallback if a Bubbles v2 widget lags in P2.2:** v1
   `github.com/charmbracelet/bubbletea v1.3.10` + `lipgloss v1.1.0` + `bubbles v1.0.0`.
2. **`tea.Msg` is not `any`.** In v2 it is an alias for a **method-less `ultraviolet.Event`**.
   Verified empirically: `*tea.Program` satisfies `Send(tea.Msg)` but **not** `Send(any)`. So the
   `programSender` seam references `tea.Msg` (the stub implements `Send(tea.Msg)`; any struct is a
   valid Msg, so the seam message types stay plain values). The terminal-free testing the plan
   wanted works exactly as intended.

## Next: P2.2 — the Model/Update/View skeleton (the first pane)

Build the disciplined Bubble Tea split under `internal/tui` (plan §4 P2.2; TDD §5 TUI row):

- A `Model` with explicit `state ∈ {idle, running, awaitingApproval, errored}`, an input box
  (Bubbles `textarea`), a scrollback (Bubbles `viewport`), and a status line (model · endpoint ·
  mode · bypass · turn counter · spinner when running). `Init`/`Update`/`View`.
- **`Run` finally does its job:** create the `Model` + `tea.NewProgram(model)`, **call
  `br.Bind(program)` *before* `program.Run()`** (so the late-bound sink/approver reach the live
  program), then run. This is the one wiring step P2.1 deliberately left as a placeholder.
- `Update` folds the five C1–C4 messages + keypresses; a keypress→submit launches the worker via
  `startExchange` and stores the returned `CancelFunc`; **submit while `running` is a no-op**
  (the single-worker invariant the whole seam relies on). **No agent logic** (C5).
- **Make the Charm v2-vs-v1 call here** (plan §5 / phase-0 §1.1) at the first real widget need and
  record it in the P2.2 commit. Add `lipgloss/v2 v2.0.4` + `bubbles/v2 v2.1.0` (or the v1 trio).
  Note the v2 deps will be on the **`charm.land/…`** path too — check before `go get`.
- **Acceptance:** `Update` unit-tested by feeding synthetic Msgs and asserting state transitions +
  `View` substrings (golden per `testing.go.md`); `WindowSizeMsg` reflows without panic; submit
