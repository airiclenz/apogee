## 2. The four decisions (the agenda)

For each: **where it's specified** (don't re-paste — read it), **the plan's current default**, **what

### Decision A — Charm TUI stack: pin the **v2** line
- **Spec:** plan §1.1. Default = `bubbletea v2.0.7` / `lipgloss v2.0.4` / `bubbles v2.1.0`; documented
  fallback = the v1 trio (`v1.3.10` / `v1.1.0` / `v1.0.0`).
- **Stakes: lowest of the four.** Adds **zero deps now** (first use is Phase 2) and is explicitly

- **Ask the owner:** any known reason to start on v1 instead — e.g. a specific community widget that
  only ships for v1? If not → **confirm v2 as the target, v1 as the documented fallback.**
- **On confirm:** nothing to build; the pin already reads as decided. No ADR (dep pins aren't ADR-tracked).

### Decision B — MCP `go-sdk` maturity verdict
- **Spec:** plan §1.2 (and the broad plan's §5 risk table / §6). `github.com/modelcontextprotocol/go-sdk`
  is at `v1.6.1` — past `v1.0.0` GA, 6 stable minors. Default verdict = **stay on the official SDK pinned
  `v1.6.x`; drop `mark3labs/mcp-go` from the active fallback set**; Phase 3 re-confirms at entry.
- **Ask the owner:** OK to **retire** the mark3labs fallback from the active set (keep it only as a
  one-line break-glass note if the official SDK regresses)? First real use is Phase 3, so this is also
  re-confirmed then.
- **On confirm:** propagate to the **broad** [`implementation-plan-apogee-merge.md`](../plans/implementation-plan-apogee-merge.md)
  §5 risk row + §6 (mark the "new SDK / evaluate two" risk as substantially retired). Pure doc edit.

### Decision C — the hermetic-Upstream `responder` seam *(the one real design call)*
- **Spec:** plan §3, "P0.6 — … The hermetic-Upstream seam (the one design call P0.6 must make)".
- **The call:** `Step` needs an Upstream response, but the real provider is Phase 1 and the **public**
  `Config` exposes only `Endpoint`/`Model`. Resolution = build the minimal loop over an **unexported
  `internal/agent` seam** (sketch: `type responder interface { respond(ctx, *request) (rawResponse, error) }`),
  with a **deterministic fake** in test code; the real HTTP provider implements the *same internal seam*
  in Phase 1. Public surface stays untouched; tests reach `internal/` because they're in-module.
- **Ask the owner to confirm:** (1) provider injection stays **internal-only** — no public provider seam
  added for a test; (2) the seam lives in `internal/agent`, unexported; (3) the names (`responder` /
  `respond` / `request` / `rawResponse`) are *sketch* — fine to refine when implemented. The plan argues
  internal-only is the correct architecture anyway (recommend confirm).
- **On confirm:** this is the green light for the P0.6c loop body. Consider a short design note (TDD §5
  agent/provider row) recording the seam; an ADR is likely overkill.

### Decision D — P0.6 scope (keep the slice throwaway-thin)
- **Spec:** plan §3, P0.6 "Scope discipline — what is and isn't built" (in/out table) + sub-tasks **P0.6a–e**.
- **In:** `New`/`Config` validation (Auto-gate ADR 0004 + ordering-cycle ADR 0003 + require `Events`);
  one conversation-state value + versioned JSON `Snapshot`/`Resume`; **one** non-streaming Turn
  (`Submit`→`Step`→build `Request`→ask Upstream→`MessageEvent`→quiescent boundary); `ctx` cancel →
  `StatusCancelled` + resumable; recover-at-extension-boundary (panicking hook → `ErrorEvent`, clean
  boundary); `MechanismRegistry.AddExperimental` → pre-request hook → `MechanismFiredEvent`.
- **Out:** streaming, real HTTP client, concrete Session schema/migration, real tool dispatch/approval,
  all-format tool-call parsing, sub-agents/modes/confinement backends, descriptor-carrying catalogued
  Mechanisms / topo-order beyond cycle-check.
- **Ask the owner:** is this slice thin enough and complete enough? Any item to pull in or push out —
  e.g. is single-Turn / non-streaming acceptable for the capstone (plan says cancel happens *mid-`respond`*,
  not mid-stream)? Recommend **confirm as specified.**
- **On confirm:** P0.6a–e + the P0.6e test become the implementation worklist.
