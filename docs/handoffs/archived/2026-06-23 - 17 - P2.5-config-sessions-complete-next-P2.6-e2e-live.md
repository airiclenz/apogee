## What P2.5 built that P2.6 builds on

- **Config resolution** lives in `cmd/apogee/config.go` (`resolveSettings` + flag/env/file layers,
  `applyConfig` wired in `RunE`). The live/e2e run can set `--endpoint`/`--model`/`--mode` via flag,
  `APOGEE_*` env, **or** `~/.apogee/config.yaml` — all four tiers resolve (flag > env > file >
  default).
- **First-run auto-seed:** `cmd/apogee/defaults/config.yaml` is `//go:embed`-ed and written to
  `<home>/config.yaml` on first run (never overwritten). **Gotcha for P2.6 tests:** a real run
  writes into `~/.apogee` — keep tests hermetic by passing `--config <t.TempDir()>` (as
  `TestRootCommandExecuteCleanQuit` now does). The template is behaviour-neutral (all commented),
  so it never injects values into a test/live run.
- **The saver seam:** `internal/session.Store` (`Save(domain.Session) (path, error)`) +
  `tui.Options.Save func(domain.Session) error`. The model calls `eng.Snapshot()` + `Save` **only
  on a clean quit at idle/errored** (no worker owns the single-goroutine Agent). **A `ctrl+c`
  mid-run cancels and quits WITHOUT saving** — so the P2.6 e2e must drive the Exchange to
  completion (idle) before quitting if it wants to assert a snapshot was written. `--resume`
  reconstruction is already proven (`TestSessionSaverRoundTripsThroughResume` + the wire-layer
  future-version test).

## Next: P2.6 — end-to-end acceptance (+ the Phase-1 live confirmation)

Plan §4 P2.6 is the spec. Two parts:

1. **Hermetic `httptest` e2e** (the reproducible proof, mirroring P1.7's `coreagent` pattern): a
   scripted OpenAI-compatible `httptest` model drives the **real** Agent **through the TUI's seam**
   (`teaSink` + `uiApprover` + worker) — stream tokens → `write_file` tool call → approve (scripted
   key or an auto-approver) → file lands in a `t.TempDir()` workspace → transcript shows tokens →
   call → result → final message. No terminal required — drive via `teatest` or the existing
   `programSender` stub (see `internal/tui/seam_test.go`'s `stubProgram`). **Then** snapshot-on-quit
   → `--resume` → continue at the right Turn closes the P2.5 acceptance end-to-end.
2. **Live-model run** from the host: load a tool-capable model via the llama-launcher MCP control
   (`http://192.168.61.1:7331/mcp`), point `--endpoint http://192.168.64.1:1111` at it, hold a real
   file-edit conversation, watch tools run, approve the write. This **closes the open Phase-1
   live file-edit eval** through the product UI. Needs a TTY (use the `run`/`verify` skills /
   `manage-llm-server`).

This is the **last Phase-2 task**; the seam + skeleton + fold + approval + config + sessions are
all in place for it. No engine change is expected (if the TUI seems to need new engine behaviour,
that is a Phase-1-surface gap to fix in the library — plan §6).
