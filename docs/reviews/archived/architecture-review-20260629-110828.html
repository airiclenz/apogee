Architecture review — apogee

2026-06-29 · deepening opportunities

apogee is a terminal coding agent that owns the whole loop for a small local LLM. The packages are mostly well-shaped — the Confiner seam (three real adapters), the worker/bridge threading model, and the tool catalogue all earn their keep. What follows are the places where an interface is nearly as complex as its implementation (shallow), where behaviour is smeared across files with no single test surface (no locality), or where a deep module exists but no caller crosses its seam (dormant leverage).

Strong in-process #1

Collapse the dispatch decision into one Resolution

internal/agent/dispatch.go · internal/agent/disposition.go · internal/security/guard.go

Before — the verdict is smeared
flowchart TD
  C([tool call]) --> G[guards.PreExecute]
  G -->|Refuse| X1((refuse))
  G --> SA{isSubAgentCall?}
  SA -->|yes| REC[runSubAgent]
  SA -->|no| D[dispose: mode x class x caps]
  D --> UP{{Tier-2 forces gate}}
  D --> EX[executeTool]
  EX --> CF{{install Confinement box}}
  EX --> RT{{runtime: cannot confine -> demote to gate}}
  UP -.override.-> EX
  RT -.override.-> AP[approve fallback]
  classDef leak stroke:#dc2626,stroke-width:2px,fill:#fef2f2,color:#7f1d1d;
  classDef tbl fill:#eef2ff,stroke:#6366f1,color:#3730a3;
  class UP,RT,CF leak
  class D tbl

dispose() claims to be "the single place the autonomy ladder becomes code," but two verdict-changing rules — the Tier-2 force-gate and the runtime confine-unavailable demote — live in dispatch.go prose, not the table.

After — one deep module, one verdict
resolve(mode, call, guard, caps)
→ Resolution
ladder × class Tier-2 force caps / confine runtime demote box build refuse reason
Run · Confine(box) · Gate(reason, forcible) · Refuse(why)

dispatch.go shrinks to: compute the Resolution, then execute it. Every override becomes a field of the verdict, not a branch in the executor.

Problem

The "will this call run, confine, gate, or refuse?" decision has no single seam — three of its rules sit outside the disposition table.

Solution

Return a complete Resolution verdict (including force-gate, confine-box, and the runtime-demote rule) so dispatch only executes it.

Wins
  • ▪ locality: the whole safety matrix in one place
  • ▪ interface shrinks to one verdict type
  • ▪ one test surface: guard × mode × caps together
  • ▪ dispatch becomes a thin executor
  • ▪ deletion test: complexity concentrates, not moves
  • ▪ the table stops lying about being "single"

After — the loop crosses the seam

One config field (ToolCalling) selects the adapter; the loop strips channels and parses through the same interface. The deep module stops being dormant.

Problem

The format-aware parser is built and tested but the loop never crosses its seam — production parsing is native-only and thinking/harmony channels leak into content.

Solution

Add a ToolCalling field to domain.Config and call the factory + strippers at the parse seam in loop.go.

Wins
  • ▪ leverage: one interface, four formats
  • ▪ channel text stops leaking into replies
  • ▪ oracle-tested module becomes load-bearing
  • ▪ small-model coverage stops being native-only
  • ▪ deletion test: today it deletes for free
  • ▪ loop test surface gains text-format cases
Note — this is a missing wire, not a redesign. Confirm it is genuinely deferred work (a Phase gap) rather than intentionally-off behaviour before treating it as friction.

Lift the chat input out of the god-Model

The five indigo bands are one coherent concept — the chat input — but they are loose fields on Model. acceptAutocomplete mutates m.input across a file boundary; it is only testable through the full Update loop.

After — Model coordinates; input is deep

One module, one small interface, tested without a fake engine or a terminal.

Problem

The input mini-language (parse, autocomplete, skills, file-refs, mouse-select) is spread across the 1185-line Model as loose fields, so its real logic is only reachable through Update.

Solution

Group it behind a promptEditor module the Model embeds and delegates to, with a small Submit-shaped interface.

Wins
  • ▪ locality: input logic in one module
  • ▪ mini-language tested without a fake engine
  • ▪ Model returns toward "thin renderer"
  • ▪ accept/attach stop leaking across files
  • ▪ file-walk I/O hides behind one seam
Tension with the framework — Bubble Tea favours a flat value-Model. The deepening is the input cluster, not every concern; stats and transcript-cache are genuinely fused to View/layout and should stay.

Worth exploring in-process #4

Make Fired() real, or delete the seam

internal/domain/hooks.go:217 · internal/domain/hookview.go:16 · internal/agent/hookrun.go

Before — a seam with zero adapters
Mechanism reads view.Fired(id)
fired map → nil
never populated · always returns 0

The interface promises cross-Mechanism coordination ("fire only if X hasn't"). The backing map is never filled, so every author who trusts it codes a gate that never trips. One declared seam, zero adapters.

After — pick one
Deepen

Thread the per-Session fire counts (already emitted as MechanismFiredEvent) into View() so Fired() returns the truth.

— or —
Remove

Delete the method until a Mechanism needs it. A seam with no adapter is a hypothetical seam; one that lies is worse than absent.

Problem

A documented interface method on the hook working-value always returns 0 — its backing state is never wired, so it silently misleads Mechanism authors.

Solution

Either populate the fire-count map through the engine seam (deepen) or drop the method (no adapter justifies it yet).

Wins
  • ▪ the interface stops lying
  • ▪ testable coordination, or none promised
  • ▪ one adapter ⇒ a real seam
  • ▪ smaller interface for hook authors

Top recommendation

Start with #1 — collapse the dispatch decision into one Resolution

It is the load-bearing safety invariant of the whole agent, and it is the one place where the code's own claim — "the single place the autonomy ladder becomes code" — is false: the Tier-2 force-gate and the runtime confine-unavailable demote live in the executor, not the table. Deepening it concentrates the entire mode × class × caps × guard matrix behind one verdict, gives it a single test surface, and shrinks dispatch.go to an executor. #2 is close behind and cheaper — wiring the dormant parser seam is mostly connection, not redesign — so it is the natural second move once the decision core is clean.
